Always edit the repo copy first, then deploy.

**Why:** the repo is the single source of truth across bazzite + Deck +
any future host. Editing the live file in place means (a) the change
isn't reviewable, (b) the next deploy from the repo overwrites it
silently, and (c) the live file can be lost (Nexus rewrites
`InputBinds.json` on GW2 shutdown — an in-place edit while Nexus has its
in-memory copy can be clobbered). The project's `CLAUDE.md` already
states this explicitly ("Repo-first configuration: never edit live
configs directly"); this memory exists because I edited a live file
once and the user had to remind me of the rule.

**How to apply:**

1. If the file isn't already tracked in the repo, `scp` it from the
   host to the conventional location (e.g.
   `configs/gw2-keybinds/nexus-inputbinds.json`,
   `configs/bazzite/*`, `configs/steam-controller/*`).
2. Make the edit in the repo file.
3. Commit + push.
4. Deploy to all reachable hosts. For DLLs use the dedicated deploy
   scripts (`scripts/deploy-mystic-{ai,clicker}-dll.sh`); for static
   files use the existing pattern (`scp` with GW2-closed pre-flight, or
   `ssh ... 'cat > /path'` for paths with spaces).
5. Verify on the live host (hash, mtime, or service status).

Applies to:

- Nexus addon configs (`InputBinds.json`, `AddonConfig.json`,
  `GameBinds.xml`)
- Steam Input controller VDFs (always validate per
  [[vdf-editing-rules]] golden rules)
- bazzite host scripts (`~/scripts/*.py`, `~/bin/*`)
- systemd user units (`~/.config/systemd/user/*.service`,
  `*.timer`)
- daemon config templates (`gw2-claude-setup.sh` etc. — the keys go to
  `~/.config/gw2-claude/config.env` which is local-only, but the
  *template* and the *script that creates it* live in the repo)
- Deck-equivalent paths for the same files

Exceptions are explicit `~/.config/gw2-claude/config.env` actual values
(API keys — local-only by design, never in the repo) and per-host
`mystic-ai-<W>x<H>.cfg` resolution configs (deleted/regenerated by the
DLL, not source-controlled).
